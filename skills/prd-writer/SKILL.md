# Skill: PRD Writer

> 辅助撰写需求文档，从用户痛点出发，产出结构化的需求文档（含用户故事和 GIVEN/WHEN/THEN 验收条件）。

## 触发条件

- 用户要求写需求文档、产品需求或 PRD
- 用户讨论产品定位、目标用户、功能需求
- 用户提到 "Phase 1"、"需求层"、"WHY"
- 当前处于项目的需求分析阶段

## 核心能力

1. 引导用户梳理产品定位和目标用户
2. 提炼用户痛点，结构化为需求条目
3. 编写用户故事和 GIVEN/WHEN/THEN 验收条件
4. 排列功能优先级
5. 识别约束和"不做"清单
6. 生成符合 OpenLogos 规范的需求文档

## 执行步骤

### Step 1: 理解产品定位

与用户确认以下核心问题（如果信息不足则主动追问）：

- **一句话定位**：这个产品是什么？为谁？解决什么问题？
- **目标用户画像**：具体到可以描述一个真实的人
- **核心目标**：产品要达成什么，用什么指标衡量成功

### Step 2: 提炼用户痛点

引导用户从以下维度梳理痛点：

- 用户当前是怎么做的？（现状）
- 做的过程中遇到什么困难？（痛点）
- 困难导致了什么后果？（影响）
- 用户期望怎么被解决？（期望）

**每条痛点必须有因果链**：因为 [原因] → 导致 [痛点] → 造成 [后果]

### Step 3: 归纳功能需求

将痛点转化为功能需求列表：

- 每条功能需求对应一个或多个痛点
- 标注优先级：P0（核心） / P1（重要） / P2（可选）
- 明确功能边界：做什么 + 不做什么

### Step 4: 编写用户故事和验收条件

为每个 P0 和 P1 功能编写用户故事：

```markdown
### 用户故事：[故事名称]

**作为** [用户角色]，
**我想要** [目标/行为]，
**以便** [获得的价值]。

#### 验收条件

##### 场景：[正常场景名称]
- **GIVEN** [初始条件/上下文]
- **WHEN** [用户执行的操作]
- **THEN** [期望的系统行为]
- **AND** [附加期望结果]

##### 场景：[异常场景名称]
- **GIVEN** [初始条件]
- **WHEN** [用户执行的操作] + [异常触发条件]
- **THEN** [期望的错误处理行为]
```

**验收条件编写原则**：

- 每个用户故事至少 1 个正常场景 + 1 个异常场景
- GIVEN 描述初始状态，要具体到可复现
- WHEN 描述用户操作，要精确到按钮级别
- THEN 描述系统行为，要具体到可验证
- 避免模糊词汇："快速"、"友好"、"合理" → 量化为具体指标

### Step 5: 识别约束和边界

- **技术约束**：技术栈限制、第三方服务限制
- **资源约束**：团队规模、时间窗口
- **"不做"清单**：明确列出本阶段不做的功能，避免范围蠕变

### Step 6: 组装需求文档

按标准结构输出完整文档：

```markdown
# [产品名称] 需求文档

> 最后更新：[日期]

## 一、产品背景与目标
### 1.1 产品定位
### 1.2 核心目标
### 1.3 目标用户画像

## 二、用户需求分析
### 2.1 用户痛点
### 2.2 用户期望
### 2.3 用户决策路径

## 三、功能需求概述
### 3.1 核心功能列表
### 3.2 用户故事与验收条件
### 3.3 功能优先级矩阵
### 3.4 非功能需求

## 四、约束与边界
### 4.1 技术约束
### 4.2 资源与时间约束
### 4.3 "不做"清单
```

## 输出规范

- 文件格式：Markdown
- 存放位置：`resources/prd/1-product-requirements/`
- 文件命名：`{序号}-{英文名}.md`，如 `01-requirements.md`
- 每条需求必须可追溯到用户痛点
- P0/P1 功能必须有用户故事 + GIVEN/WHEN/THEN

## 实践经验

- **先宽后窄**：第一轮尽可能多地收集需求，然后在优先级排序时砍掉非核心的
- **用户故事不是功能描述**：重点在"谁"+"为什么"，而不是"系统怎么做"
- **验收条件是需求的精确表达**：如果写不出 GIVEN/WHEN/THEN，说明需求还没想清楚
- **异常场景同等重要**：用户不可能总是走正常路径，异常处理往往决定产品体验
- **"不做"清单是最难写的**：克制是产品经理最重要的能力，明确不做什么比决定做什么更难
- **需求文档是活文档**：随着产品演进持续更新，通过 Delta 变更管理跟踪每次变化
