# Skill: Test Orchestrator

> [WIP] 基于业务场景和时序图设计 API 编排测试用例，覆盖正常/异常/边界场景，作为代码验收标准。

## 触发条件

- 用户要求设计测试用例或 API 编排
- 用户提到 "Phase 3 Step 3"、"测试先行"、"API 编排"
- 已有场景时序图和 API 规格，需要设计测试编排
- 用户需要验收已部署的代码

## 核心能力

1. 从时序图和 API YAML 设计正常流程编排
2. 基于异常用例（EX-N.M）设计异常流程编排
3. 设计边界用例（合法但非主路径的变体）
4. 定义变量提取和传递机制
5. 执行编排并验证结果

## 执行步骤

> 详细步骤待完善。核心流程：
>
> 1. 读取场景时序图，提取 API 调用链
> 2. 设计正常流程编排（每步对应时序图的一个 Step）
> 3. 为每个 EX 异常用例设计异常编排
> 4. 识别边界用例，设计边界编排
> 5. 输出可执行的编排 JSON

## 输出规范

- 文件格式：JSON
- 存放位置：`resources/scenario/`
- 按场景分文件：`user-auth.json`、`payment-flow.json`
- 编排中的每一步对应时序图的 Step 编号

## 实践经验

- **正常编排是骨架**：先完成正常流程编排，确保主路径可以跑通
- **异常编排是保障**：每个外部调用至少 1 个异常编排
- **变量传递**：前一步的响应中提取变量（如 token、user_id），传给后续步骤
- **测试数据**：编排开始前准备测试数据，结束后清理，保证幂等性
- **并发测试**：关键场景需要考虑并发情况（如：两人同时注册同一邮箱）
