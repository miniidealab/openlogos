# 研发工作流定义

> 版本：0.1.0
>
> 本文档定义 OpenLogos 的三层推进工作流：WHY → WHAT → HOW。这是方法论的核心骨架，所有 Skills 和工具都围绕这个工作流设计。

## 总览

```
Phase 1: WHY — 为什么做
├── 用户研究 → 竞品分析 → 需求归纳 → 优先级排序
├── 产出：需求文档（含用户故事 + GIVEN/WHEN/THEN 验收条件）
└── 质量门禁 → Gate 1

Phase 2: WHAT — 做什么
├── 信息架构 → 功能规格 → HTML 原型 → 设计规范
├── 产出：产品设计文档 + HTML 原型（含功能验收条件）
└── 质量门禁 → Gate 2

Phase 3: HOW — 如何做
├── Step 1: 场景建模 → 时序图 → API 浮现
├── Step 2: API 细节设计 → DB 推导
├── Step 3: API 编排设计（测试先行）
├── Step 4: AI 驱动代码生成 → 部署测试环境
├── Step 5: API 编排验收
└── 质量门禁 → Gate 3.1 ~ Gate 3.5
```

## Phase 1: WHY — 需求层

### 目标

搞清楚要解决什么问题。产出清晰、无歧义的需求文档。

### 工作内容

1. **用户研究**：理解目标用户、痛点、使用场景
2. **竞品分析**：了解已有方案的优劣势
3. **需求归纳**：将散乱的想法结构化为需求条目
4. **优先级排序**：区分核心/重要/可选需求
5. **用户故事编写**：为核心功能编写 GIVEN/WHEN/THEN 验收条件

### 产出物

需求文档，标准结构：

- 产品背景与目标（定位、目标、用户画像）
- 用户需求分析（痛点、期望、决策路径）
- 功能需求概述（功能列表 + 用户故事 + 验收条件 + 优先级矩阵）
- 约束与边界（技术/资源/时间约束 + "不做"清单）

### 用户故事格式

```markdown
### 用户故事：[故事名称]

**作为** [用户角色]，
**我想要** [目标/行为]，
**以便** [获得的价值]。

#### 验收条件

##### 场景：[正常场景]
- **GIVEN** [初始条件]
- **WHEN** [用户操作]
- **THEN** [期望结果]
- **AND** [附加结果]

##### 场景：[异常场景]
- **GIVEN** [初始条件]
- **WHEN** [操作] + [异常触发]
- **THEN** [错误处理行为]
```

### Gate 1 检查项

- [ ] 每条核心需求有用户痛点支撑
- [ ] 每个核心功能有用户故事 + GIVEN/WHEN/THEN 验收条件
- [ ] 目标用户画像具体到可描述一个真实的人
- [ ] 功能优先级已排序
- [ ] "不做"清单明确

## Phase 2: WHAT — 产品设计层

### 目标

设计具体的解决方案满足 Phase 1 的需求。产出产品设计文档和可交互的 HTML 原型。

### 工作内容

1. **信息架构设计**：产品结构、导航层级、内容组织
2. **功能规格撰写**：每个功能的详细描述 + GIVEN/WHEN/THEN 验收条件
3. **产品原型设计**：使用 AI 直接生成 HTML 页面作为产品原型
4. **设计规范制定**：全局 UI 规范

### 产出物

- 产品设计文档（信息架构 + 功能规格 + 需求追溯矩阵）
- HTML 产品原型
- 全局设计规范

### Gate 2 检查项

- [ ] 每个核心功能有详细规格 + GIVEN/WHEN/THEN 验收条件
- [ ] 所有核心页面有 HTML 原型
- [ ] 需求追溯矩阵完整
- [ ] 异常情况已考虑（错误/空/加载状态）

## Phase 3: HOW — 技术实现层

### Step 1: 场景建模

从业务场景出发，绘制 Mermaid 时序图。时序图中的跨系统调用箭头就是需要的 API。

**时序图规范**：
- 每个箭头带 `Step N:` 编号前缀
- 每个箭头附一行行为描述
- 参与方明确标注系统组件

**Gate 3.1**：所有核心场景时序图完成，API 端点清晰可见。

### Step 2: API 细节设计 → DB 推导

基于时序图浮现的 API 端点，设计详细规格（OpenAPI 3.0 YAML）。API 的请求/响应结构确定后，DB 表结构自然推导。

**Gate 3.2**：API YAML 和 DB DDL 完成，相互一致。

### Step 3: API 编排设计（测试先行）

在写代码之前，先为每个场景设计 API 编排测试用例：

- **正常流程编排**：主路径的 API 调用链
- **异常流程编排**：`EX-{步骤编号}.{序号}` 格式
- **边界用例**：合法但非主路径的变体

**Gate 3.3**：编排覆盖所有正常流程 + 核心异常流程。

### Step 4: AI 驱动代码生成

AI 面前已有完整上下文（原型 + 场景 + API + DB + 编排），此时生成的代码质量远高于直接写代码。按场景逐个生成、逐个验证。

**Gate 3.4**：代码已审核，已部署测试环境。

### Step 5: API 编排验收

运行 Step 3 设计的编排来验证代码。全部通过 = 场景开发完成。

**Gate 3.5**：所有编排全部通过。

## 验收条件的三级递进

| 层次 | 粒度 | 关注点 |
|------|------|--------|
| Phase 1 用户故事 | 业务行为 | 用户想要什么、为什么 |
| Phase 2 功能验收 | 交互细节 | 界面怎么响应、状态怎么变化 |
| Phase 3 API 编排 | 技术验证 | 接口返回什么、数据库写了什么 |

三层验收条件逐层细化，最终在 Phase 3 的 API 编排中变成可自动执行的测试。

## 迭代规则

功能迭代**必须**按同样的分层工作流推进，使用 Delta 变更管理（详见 [change-management.md](./change-management.md)）。不允许跳过中间环节直接改代码。
