## 粒度检查结论

Phase 1 的场景定义粒度合理，每个场景均有明确的用户目标（不是单纯的 CRUD 操作）：

| 场景 | 用户目标 | 粒度评估 |
|------|---------|---------|
| S01 | 建立个人任务空间并开始使用 | ✅ 合理 |
| S02 | 快速处理当日任务（登录→增删改查完整闭环） | ✅ 合理，按交互拆为子场景 |
| S03 | 将混乱任务按类型归整 | ✅ 合理 |
| S04 | 自主管理账号安全 | ✅ 合理 |

## 场景依赖关系

```
S01（注册）
  └─► S02.1（登录）
        ├─► S02.2（创建任务）
        ├─► S02.3a（完成任务）
        ├─► S02.3b（编辑任务）
        ├─► S02.3c（删除任务）
        ├─► S03.1（创建分类）—— 分类创建后才能在 S02.2 中选择
        ├─► S03.2（筛选任务）
        ├─► S03.3（删除分类）
        └─► S04（修改密码）
```

- S01 是所有场景的前置条件（必须先有账号）
- S03.1 是 S02.2 分类选择功能的前置（无分类时任务只能为"无分类"）
- S02.x / S03.x / S04 在登录后可独立触发，无顺序依赖

## Tauri 命令清单（由时序图汇总）

以下命令由时序图中的 `invoke(...)` 调用汇总，是下一阶段 API 设计的直接输入：

| 命令名 | 所属场景 | 入参 | 返回 |
|--------|---------|------|------|
| `check_has_user` | S01, S02.1 | — | `{ has_user: bool }` |
| `register` | S01 | `{ username, password }` | `{ user_id, username }` |
| `login` | S02.1 | `{ username, password }` | `{ user_id, username, tasks[] }` |
| `create_task` | S02.2 | `{ user_id, name, category_id?, note? }` | Task 记录 |
| `update_task_status` | S02.3a | `{ task_id, done }` | `{ task_id, done }` |
| `update_task` | S02.3b | `{ task_id, name, category_id?, note? }` | Task 记录 |
| `delete_task` | S02.3c | `{ task_id }` | `{ success }` |
| `create_category` | S03.1 | `{ user_id, name }` | Category 记录 |
| `delete_category` | S03.3 | `{ category_id }` | `{ success }` |
| `change_password` | S04 | `{ user_id, current_password, new_password }` | `{ success }` |

> S03.2（按分类筛选）为纯前端操作，无 Tauri 命令。
