# 轻记账 App 需求文档

> 最后更新：2026-04-09（补充密码锁场景 + Electron 技术栈）

## 一、产品背景与目标

### 1.1 产品定位

面向职场人群的轻量记账工具——一步记录开销，分类分时段查询消费情况，替代频繁翻看支付宝/微信账单的碎片化习惯。

### 1.2 核心目标

- **北极星指标**：用户次日留存率 ≥ 40%（衡量"简洁好用"）
- **数据准备成功标准**：用户能完成至少 10 条记账记录，且能成功查看统计报表

### 1.3 目标用户画像

| 属性 | 描述 |
|------|------|
| 年龄段 | 24-50 岁在职人群 |
| 收入范围 | 月收入 5000-30000 元 |
| 现有记账习惯 | 无主动记账，仅被动查看支付宝/微信账单 |
| 核心诉求 | 快速记录、知道钱花哪儿、了解消费趋势 |
| 痛点 | 现有工具操作复杂，看完账单想不起分类汇总 |

---

## 二、用户痛点分析

### P01: 消费后想不起钱花哪儿

因为支付宝/微信账单只有金额和商家 → 导致消费记录分散、无法分类汇总 → 造成月底复盘时一脸茫然

### P02: 想记账但操作太麻烦坚持不下来

因为现有记账 App 步骤多（选分类→输金额→选日期→填备注→保存） → 导致每次记账像做任务 → 造成最多坚持一周就放弃

### P03: 月底手动统计各类别支出太麻烦

因为支付宝/微信没有分类汇总功能 → 导致需要逐条查看手动加总 → 造成月底复盘成本极高

### P04: 桌面多人使用，数据无保护

因为是桌面应用且可能多人共用一台电脑 → 导致任何人都能打开 App 查看/修改所有消费记录 → 造成隐私泄露风险

## 三、场景总览

| 编号 | 场景名称 | 触发条件 | 关联痛点 | 优先级 |
|------|---------|---------|---------|--------|
| S01 | 快速记账 | 用户打开 App → 选择分类 → 输入金额 → 确认保存 | P02 | P0 |

## 四、核心场景详述

### S01: 快速记账

- **触发条件**：用户打开 App 记账入口
- **用户价值**：3 秒内完成一笔记账，无心理负担（← P02）
- **优先级**：P0
- **主路径**：打开 App → 出现默认记账界面（分类已展示）→ 输入金额 → 点击「记」按钮 → 显示保存成功

#### 验收条件

##### 正常：完整记账流程
- **GIVEN** 用户已打开 App 记账界面，默认显示所有分类
- **WHEN** 用户选择「餐饮」分类，输入「35.5」，点击「记」
- **THEN** 显示「已记录」，分类默认停留在「餐饮」，金额清空，等待下一笔

##### 正常：修改默认分类
- **GIVEN** 用户刚完成一笔记账，当前分类为「餐饮」
- **WHEN** 用户选择「交通」分类，输入「12」，点击「记」
- **THEN** 显示「已记录」，下次打开 App 默认分类变为「交通」

##### 异常：未选择分类
- **GIVEN** 用户已打开 App 记账界面
- **WHEN** 用户未选择分类，直接输入金额「100」，点击「记」
- **THEN** 按钮置灰不可点击，或点击后提示「请选择分类」

##### 异常：金额为空或为 0
- **GIVEN** 用户已打开 App 记账界面，已选择「购物」分类
- **WHEN** 用户输入金额为空或「0」，点击「记」
- **THEN** 按钮置灰不可点击，或点击后提示「请输入有效金额」

##### 异常：金额格式错误
- **GIVEN** 用户已打开 App 记账界面，已选择「娱乐」分类
- **WHEN** 用户输入「abc」或「-50」，点击「记」
- **THEN** 显示「请输入有效金额」，不保存

### S02: 查看统计报表

- **触发条件**：用户点击「统计」入口进入统计页面
- **用户价值**：快速了解某段时间内各类别的支出汇总（← P01, P03）
- **优先级**：P0
- **主路径**：进入统计页 → 选择时间范围（如 4 月） → 选择分类（可选，默认全部） → 查看柱状图/列表汇总

#### 验收条件

##### 正常：查看整月分类汇总
- **GIVEN** 用户有 4 月的消费记录（餐饮 800 元、交通 200 元、购物 1500 元）
- **WHEN** 用户进入统计页，选择「4 月」，分类选择「全部」
- **THEN** 显示柱状图：餐饮 800、交通 200、购物 1500，总计 2500 元

##### 正常：查看单月单分类明细
- **GIVEN** 用户有 4 月的餐饮消费记录（早餐 300、午餐 400、晚餐 100）
- **WHEN** 用户进入统计页，选择「4 月」，分类选择「餐饮」
- **THEN** 显示餐饮分类下的消费列表，总计 800 元

##### 正常：查看周统计
- **GIVEN** 用户有本周消费记录
- **WHEN** 用户进入统计页，选择「本周」
- **THEN** 显示本周每日消费柱状图 + 各类别汇总

##### 正常：时间范围筛选
- **GIVEN** 用户有 3 月、4 月、5 月的消费记录
- **WHEN** 用户进入统计页，选择「3月1日 - 4月30日」
- **THEN** 仅显示 3-4 月的消费汇总

##### 异常：时间范围无数据
- **GIVEN** 用户选择的时间范围内没有任何消费记录
- **WHEN** 用户选择「1 月」但 1 月没有记录
- **THEN** 显示空状态图，文字提示「该时间段暂无记录，开始记账吧」

##### 异常：分类无数据
- **GIVEN** 用户有消费记录，但选择「医疗」分类
- **WHEN** 用户选择「医疗」分类
- **THEN** 显示「医疗」分类金额为 0，不显示柱状图分项

### S03: 管理支出分类

- **触发条件**：用户在记账页或设置页进入「分类管理」
- **用户价值**：自定义符合个人消费习惯的分类（← P01）
- **优先级**：P1
- **主路径**：进入分类管理 → 新增自定义分类 / 编辑现有分类名称 / 删除不需要的分类 → 返回记账页新分类生效

#### 验收条件

##### 正常：新增自定义分类
- **GIVEN** 用户在分类管理页，当前分类列表包含「餐饮、交通、购物、娱乐、居住、医疗、其他」
- **WHEN** 用户点击「新增分类」，输入「咖啡」，点击「保存」
- **THEN** 分类列表新增「咖啡」，用户在记账页可选「咖啡」分类

##### 正常：编辑现有分类
- **GIVEN** 用户在分类管理页，存在「其他」分类
- **WHEN** 用户点击「其他」的编辑按钮，将名称改为「日常其他」，点击「保存」
- **THEN** 分类列表中「其他」变为「日常其他」，记账页同步更新

##### 正常：删除自定义分类
- **GIVEN** 用户自定义了「咖啡」分类，且该分类下有消费记录
- **WHEN** 用户点击「咖啡」的删除按钮，确认删除
- **THEN** 「咖啡」分类从列表移除，原有「咖啡」消费记录归入「其他」分类

##### 异常：新增重复分类名
- **GIVEN** 用户在分类管理页，存在「餐饮」分类
- **WHEN** 用户点击「新增分类」，输入「餐饮」，点击「保存」
- **THEN** 显示「分类名已存在，请使用其他名称」，不保存

##### 异常：编辑默认分类
- **GIVEN** 用户在分类管理页，「餐饮」是系统预设分类
- **WHEN** 用户点击「餐饮」的删除按钮
- **THEN** 删除按钮不可点击，或点击后提示「系统预设分类不可删除」

### S04: 设置密码锁

- **触发条件**：用户在设置页点击「密码锁」开关
- **用户价值**：保护记账数据隐私，防止他人查看或修改（← P04）
- **优先级**：P1
- **主路径**：进入设置页 → 点击「密码锁」开关 → 输入密码（首次）/ 输入原密码确认（关闭） → 设置成功

#### 验收条件

##### 正常：开启密码锁
- **GIVEN** 用户在设置页，密码锁当前为关闭状态
- **WHEN** 用户点击「密码锁」开关，输入密码「123456」，确认密码「123456」，点击「保存」
- **THEN** 显示「密码锁已开启」，下次打开 App 需先输入密码

##### 正常：关闭密码锁
- **GIVEN** 用户已开启密码锁，当前密码为「123456」
- **WHEN** 用户在设置页点击「密码锁」开关，输入原密码「123456」，点击「保存」
- **THEN** 显示「密码锁已关闭」，下次打开 App 无需输入密码

##### 正常：修改密码
- **GIVEN** 用户已开启密码锁，当前密码为「123456」
- **WHEN** 用户在设置页点击「修改密码」，输入原密码「123456」、新密码「654321」、确认密码「654321」，点击「保存」
- **THEN** 显示「密码修改成功」，下次解锁需输入新密码「654321」

##### 异常：两次密码不一致
- **GIVEN** 用户在设置页，开启密码锁
- **WHEN** 用户输入密码「123456」、确认密码「654321」，点击「保存」
- **THEN** 显示「两次密码不一致」，不保存

##### 异常：关闭密码锁输入错误密码
- **GIVEN** 用户已开启密码锁，当前密码为「123456」
- **WHEN** 用户在设置页点击「密码锁」开关，输入错误密码「000000」，点击「保存」
- **THEN** 显示「密码错误」，不关闭密码锁

##### 异常：忘记密码
- **GIVEN** 用户已开启密码锁，忘记了密码
- **WHEN** 用户在密码输入页面点击「忘记密码」
- **THEN** 显示确认对话框「忘记密码将清除所有数据，是否继续？」，用户确认后清除所有数据并重置 App

## 五、约束与边界

### 5.1 技术约束

| 约束项 | 说明 |
|--------|------|
| 技术栈 | Electron 桌面应用 |
| 存储 | SQLite 本地存储 |
| 通信 | HTTP API 与后端服务通信 |
| 平台 | Windows/macOS 桌面端 |

### 5.2 资源与时间约束

- 本阶段仅完成核心功能 MVP（快速记账 + 统计报表 + 分类管理 + 密码锁）
- 不追求复杂的图表样式，以表格/简单柱状图为主

### 5.3 "不做"清单

| 功能 | 原因 |
|------|------|
| 用户登录/注册 | 数据本地存储，无需账号体系 |
| 多设备同步 | MVP 阶段仅支持单设备使用 |
| 批量导入账单 | 用户明确不需要，简化功能 |
| 预算提醒 | 用户明确不需要 |
| 移动端 App | 桌面端 App |
